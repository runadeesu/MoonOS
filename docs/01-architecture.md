# Moon OS 全体アーキテクチャ

## レイヤー構成

```
┌─────────────────────────────────────────────────────────┐
│  Applications (Moon App Store apps, PWA, 3rd party)      │
├─────────────────────────────────────────────────────────┤
│  Moon Shell (GUI: Compositor, Window Manager, Launcher)  │
├─────────────────────────────────────────────────────────┤
│  System Services (Moon Daemons)                         │
│  - moond (init/service manager)                          │
│  - netd (network manager)                                 │
│  - authd (認証・権限)                                     │
│  - pkgd (パッケージ / アプリストア)                        │
│  - aid  (AIアシスタント連携)                               │
├─────────────────────────────────────────────────────────┤
│  libmoon (システムコール用 標準ライブラリ / IPC)            │
├─────────────────────────────────────────────────────────┤
│  Moon Kernel                                              │
│  - スケジューラ / プロセス管理                              │
│  - 仮想メモリ管理                                          │
│  - VFS (仮想ファイルシステム)                              │
│  - ネットワークスタック                                     │
│  - デバイスドライバフレームワーク                            │
│  - IPC / Capability ベースの権限モデル                      │
├─────────────────────────────────────────────────────────┤
│  Bootloader (UEFI, GPT対応)                               │
├─────────────────────────────────────────────────────────┤
│  Hardware (x86_64 → 将来 ARM64)                           │
└─────────────────────────────────────────────────────────┘
```

## 設計原則

1. **モノリシック・ハイブリッドカーネル**
   ドライバはカーネル空間で動作させ性能を確保しつつ、クラッシュしやすい
   ドライバ（GPU, Wi-Fi など）は将来的にユーザー空間サービスとして分離
   できるよう、ドライバ API を安定インターフェースとして設計する。

2. **Capability ベースのセキュリティモデル**
   Windows の ACL、macOS のサンドボックス、Linux の capability を参考に、
   「プロセスは明示的に渡されたケーパビリティしか使えない」を基本とする。

3. **メッセージパッシング中心の IPC**
   GUI・システムサービス間は非同期メッセージ（Moon IPC）でやりとりし、
   一部サービスがクラッシュしてもシェル全体が落ちない構造にする。

4. **ドライバのユーザー空間移行を見据えた抽象化**
   起動時は簡潔さのためカーネル内蔵ドライバから始め、後のステップで
   ユーザー空間ドライバフレームワーク（VFIO 的な仕組み）に移行する。
