# プロセス管理設計

## 基本モデル

Windows のプロセス/スレッド分離と、Linux の軽量プロセス（`clone()` に
よるスレッド実装）の両方を参考にしつつ、Moon OS では次のモデルを採用します。

- **プロセス**: 独立したアドレス空間、ケーパビリティの集合、1つ以上のスレッド
- **スレッド**: プロセス内で実行コンテキスト（レジスタ・スタック）を持つ実行単位
- **タスク**: スケジューラが扱う最小単位（= スレッド）

```rust
struct Process {
    pid: ProcessId,
    address_space: AddressSpace,
    capabilities: CapabilitySet,
    threads: Vec<ThreadId>,
    state: ProcessState,
}

struct Thread {
    tid: ThreadId,
    owner: ProcessId,
    context: CpuContext,   // 保存されたレジスタ一式
    kernel_stack: VirtAddr,
    priority: Priority,
    state: ThreadState,    // Running, Ready, Blocked, Zombie
}
```

## スケジューラ（Step 4 から段階的に実装）

1. **Step 4 初期**: ラウンドロビン、固定タイムスライス（タイマー割り込み駆動）
2. **Step 4 発展**: 優先度付きマルチレベルキュー
   （フォアグラウンド GUI スレッドを優先、バックグラウンドタスクは低優先度）
3. **将来**: ゲーム向けに、フレーム同期を意識したリアルタイム優先度クラスを追加
   （Windows の MMCSS、macOS の QoS クラスの設計思想を参考）

## コンテキストスイッチ

```
1. タイマー割り込み発生
2. 現在のスレッドのレジスタをスタックへ保存
3. スケジューラが次に実行するスレッドを選択
4. 次のスレッドのレジスタを復元
5. ページテーブル（CR3）を必要なら切り替え
6. iretq で実行再開
```

x86_64 アセンブリで実装し、Rust 側からは `extern "C"` 関数として呼び出す
薄いラッパーを用意します。

## プロセス間通信 (IPC)

メッセージパッシング方式を基本とし、以下を提供します。

- **同期メッセージ**（RPC 的な呼び出し、GUI とアプリの通信に利用）
- **非同期メッセージキュー**（イベント通知、システムサービス間通信）
- **共有メモリ領域**（大容量データ、フレームバッファ共有などパフォーマンス
  が必要な場面のみ、明示的なケーパビリティ許可の上で使用）

## Capability ベースの権限モデル

各プロセスは生成時に親から渡された「ケーパビリティ」（ファイル、
デバイス、IPC ポートへのアクセス権を表すトークン）のみを保持します。
新規プロセス生成時は、必要なケーパビリティのみを明示的に譲渡することで、
最小権限の原則を徹底します（Windows の AppContainer、macOS の
サンドボックスプロファイル、Linux の capability(7) を参考にした設計）。
